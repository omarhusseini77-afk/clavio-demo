Purpose of this document

Clavio is a data standardisation and intelligence layer. Its core purpose is to connect to a portfolio company's accounting system, standardise the financial data, and make that clean data available for two things: generated reporting and AI-powered natural language query. Both the GP and the LPs use the AI query feature - each scoped to what they are permitted to see. Reporting is one output of the layer; AI query is an equally important second output.

This plan outlines the phased path from Clavio's current working demo to a fully deployable, real-data pilot suitable for live LP, GP, and Portfolio Company users. It defines what is built with AI assistance, what requires direct human involvement, how security concerns are addressed, and how the codebase is structured for a clean handoff to a developer when needed.

The goal is to maximise what is built before a developer is engaged, reducing that developer's scope (and cost) to the minimum that legally and practically requires a named human - principally: account ownership, secret management, and the QuickBooks integration.

2
Current state (what already exists)

As of this document, the Clavio demo is a live, deployed application with the following capabilities:

  • Three isolated progressive web apps (Submit / Partner / Investor), each installable to a phone home screen with a distinct icon and URL.
  • Submit app: portfolio company submission form with AI-powered document extraction (Claude) - the ingestion layer where raw accounting data enters Clavio and is standardised.
  • Partner/GP app: revenue trends, balance-sheet signals, anomaly detection, per-company drill-down, and an AI query assistant ("Ask Clavio") that answers natural-language questions against the full standardised dataset - e.g. "why did receivables grow faster than revenue at Tecniq this quarter?"
  • Investor/LP app: capital account overview, fund performance (IRR, MOIC, TVPI, DPI), portfolio allocation, document library, and an AI query assistant scoped to fund-level and portfolio-summary data - the same intelligence layer as the GP, narrowed to LP-appropriate permissions.
  • Full bilingual support (English / French) across all three apps, including AI replies.
  • Deployed on Vercel with auto-deploy on every code commit; Supabase (hosted Postgres) for live quarter data.
  • All data currently synthetic (hardcoded demo data), no real user authentication.

Security concerns - what to worry about and what not to

The questions below are commonly raised when financial data flows through a cloud AI system. They are addressed here so this document can be shared with counterparties (legal, investors, fund administrators).

Concern Status Detail
Does Anthropic train on our data? ✓ Not a concern Under the Anthropic commercial API terms (which Clavio uses), data sent via the API is explicitly excluded from model training. This is the default - no special opt-out is needed. This is materially different from the consumer chatbot (claude.ai) terms.
Does Anthropic retain our data? ✓ Manageable Under standard commercial terms, inputs/outputs are retained for a limited operational window (trust & safety purposes) then deleted. For production use with real LP data, a Zero Data Retention (ZDR) agreement should be requested from Anthropic before go-live. This is a standard enterprise request and removes retention entirely.
Is the AI vendor reputable for financial data? ✓ Not a concern Anthropic holds SOC 2 Type II, ISO 27001, and is HIPAA-eligible. A Data Processing Agreement (DPA) is available for signing. Top-tier investment managers and fintechs already process sensitive data via the Anthropic API.
Where is data stored at rest? ✓ Configurable Supabase (the primary database) runs on AWS and allows region selection. For UK financial data, select the eu-west-2 (London) region at project creation. This is a one-time setup choice.
Can one fund's data leak to another? ✓ Handled by design Row-Level Security (RLS) in Supabase enforces data isolation at the database layer, not just the application layer. Even if there were a bug in the front-end code, an LP or portfolio company's query would physically return no rows from another fund's tables. This is Phase 3 of the build and will be audited before go-live.
Are uploaded documents secure? ✓ Handled by design Files are stored in Supabase Storage (encrypted at rest, AWS S3-backed) with the same RLS policies as the database. A document uploaded by Company A cannot be accessed by Company B. Presigned URLs expire after a short window.
API keys / secrets management ⚠ Human action required Secrets (Anthropic API key, Supabase service role key, etc.) must be held in Vercel's encrypted environment variable store and rotated by a named human with company credentials. They must never appear in code or git history. This is a setup task at Phase 0.
GDPR / data retention ⚠ Legal review required The technical infrastructure supports GDPR compliance (data deletion, audit logs, regional storage), but the policies - retention periods, data subject rights, DPA with sub-processors - must be drafted and signed off by a human with legal accountability before real client data enters the system.

Human-led Phase 0 - Accounts & Ownership ~1–2 hours, done once

All production accounts must be registered under the company's identity (not a personal account), and production secrets must be set and held by a named individual. Claude guides every step; a human clicks and owns the result.

  • Create a new Supabase project under the company account in the eu-west-2 (London) region. The current demo project is on ap-northeast-1 (Tokyo) and must not be used for real financial data.
  • Set Vercel project to production org; add environment variables (Anthropic API key, Supabase keys).
  • Register domain (e.g. app.clavio.io) and point DNS to Vercel.
  • Create Anthropic API org under company account; initiate DPA / ZDR request via Anthropic sales. Note: ZDR approval has a lead time of days to weeks - start this process before any other phase.
  • Enable Google OAuth in Supabase Auth (required for GP login); this needs a Google Cloud project registered under the company account.
  • Enable GitHub repo security (branch protection, no secrets in code, Dependabot alerts).
AI-led Phase 1 - Real Authentication ~1–2 days of build time

Replace the open demo with real login. Every user must authenticate before seeing any data. There is one login page; after signing in, each user is automatically redirected to the correct app based on their role. A portfolio company user cannot access the GP or LP views, and vice versa.

Login method is tailored to each user type:

  • Submit (portfolio company): magic link - they log in once a quarter to submit financials; no password needed for infrequent access.
  • Partner/GP (Cyril): Google OAuth ("Sign in with Google") or email + password - faster and more appropriate for regular use.
  • Investor/LP: email + password with optional Google OAuth - feels institutional and secure, appropriate for a financial portal used by professional investors.
  • Session management, protected routes, automatic redirect to login if unauthenticated.
  • Role system: user table with role field (portfolio_company / gp / lp); login redirects to correct app automatically.
  • Invite-only signup: only users explicitly invited by the GP can join - no open registration.
  • Password reset, session expiry, and "sign out" on all three apps.
AI-led Phase 2 - Real Database Schema ~1 day of build time

Design and migrate to a production-grade schema that replaces all synthetic hardcoded data. All entities - funds, companies, users, capital events, documents - become live database records.

  • Tables: funds, companies, users, memberships, quarters, capital_events, documents, ai_summaries.
  • Relationships: a user belongs to a fund in a role (lp / gp / company); a company belongs to a fund.
  • Migration scripts (versioned, repeatable).
  • Admin seed script to create the first fund, first GP user, and invite initial test LPs.
Shared Phase 3 - Per-User Data Isolation (RLS) ~1–2 days build + human sign-off

The most security-critical phase. Row-Level Security policies ensure each user's queries physically return only their own data, even in the event of a front-end bug. Built by AI; audited by a human (or a developer acting as security reviewer) before any real financial data enters the system.

  • LP policy: can only read their own membership row, their fund's capital events, and their own documents.
  • Portfolio company policy: can only read and write their own quarterly submissions.
  • GP policy: can read all records within their fund; cannot access records from other funds.
  • Test suite: automated tests that verify cross-fund data cannot be read even with a direct API call.
  • Human action: a developer or security-accountable person reviews the RLS policies and test results before go-live.
AI-led Phase 4 - File Storage ~0.5 days of build time

Replace hardcoded document lists with real uploaded files, stored securely and access-controlled to match the RLS policies.

  • Supabase Storage buckets: one per fund (encrypted, private).
  • Portfolio company upload: replaces the current demo upload zone with real file persistence.
  • GP document library: view all documents submitted by portfolio companies in the fund.
  • LP document library: view only fund reports addressed to LPs.
  • Presigned URLs (short-lived) for document download - no permanent public links.
AI-led Phase 5 - AI on Real Data ~1 day of build time

Point all AI features (extraction, Ask Clavio, document summaries) at the live database instead of synthetic data. Add a data-minimisation layer to limit what leaves Clavio's database.

  • Extraction: uploaded documents are parsed and automatically populate the quarterly submission form.
  • Ask Clavio (LP): queries answered from the user's real capital account and fund data.
  • Ask Clavio (GP): queries answered from real portfolio company financials for that fund.
  • Document summaries: live documents summarised on demand and cached in ai_summaries.
  • Data minimisation: only the specific figures needed to answer a question are sent to the AI - not the raw document with names and totals attached.
Human-led Phase 6 - QuickBooks Integration Human ownership required; AI builds the code

QuickBooks integration is the primary ingestion path - and ingestion quality is what makes everything else work. The AI query features (for both the GP and the LPs) are only as accurate as the underlying data. Clean, structured, automatically-synced accounting data is what turns "Ask Clavio" from a demo into a trusted tool. The chain is: QuickBooks → Clavio's standardised data model → reporting and AI query for GP and LP. QuickBooks is the source; Clavio's database is what everyone actually queries.

The integration code is built by AI, but the Intuit developer application must be owned and registered by a human.

  • Human action: register a developer application on the Intuit Developer Portal (developer.intuit.com) under the company account. This creates the OAuth Client ID and Secret.
  • Human action: pass Intuit's app review process (required before accessing real company data in production).
  • AI builds: OAuth 2.0 connection flow, token storage, periodic sync, mapping QuickBooks chart-of-accounts to Clavio's standardised financial schema.
  • Result: a "Connect QuickBooks" button in the Submit app; once connected, quarterly figures are pulled automatically into Clavio's data model - no manual upload needed, and both GP and LP AI query features immediately reflect the latest data.
AI-led Phase 7 - Pilot Polish ~1–2 days of build time

Round off the rough edges before real users are onboarded.

  • Email notifications: "your Q1 report is ready to review" sent to LPs and GPs.
  • PDF report export: GPs can generate a formatted quarterly report PDF for each portfolio company.
  • Admin panel: invite users, assign to funds, set roles, view submission status.
  • Error handling and loading states for all user-facing actions.
  • Basic usage logging: which features are used, how often (no personal data - for product feedback only).
6
Developer handoff scope

When a developer is engaged, their scope should be tightly defined to avoid paying for work that has already been done. The natural handoff point is after Phase 5 - the application is complete, real-auth, and running on real data. The developer's specific scope is:

Recommended developer scope

Security audit: Review the RLS policies, auth configuration, and secrets setup. Confirm they meet the standard expected for handling regulated financial data. Sign off in writing.

Account & secrets ownership: Take over (or formally document) ownership of all production accounts, rotation schedules for API keys, and incident-response runbook.

QuickBooks integration (Phase 6): Own the Intuit developer application registration and OAuth credentials. Build alongside the AI-generated sync code.

GDPR / compliance plumbing: Data retention policies, data subject access requests (DSAR) tooling, DPA counter-signatures with sub-processors.

What the developer should NOT need to build

The entire application front-end, database schema, auth system, AI integration, file storage, and deployment pipeline will already exist and be tested. The developer is hardening and owning, not building. A competent developer should be able to read the codebase in a day - it uses standard, well-documented tools (Next.js, Supabase, Vercel).
