What Clavio is

Clavio is a data standardisation and intelligence layer for private equity funds. Its core purpose is to connect to a portfolio company's accounting system (QuickBooks, Sage, Cegid, or document upload as fallback), standardise the financial data, and make that clean data available for two things: generated reporting and AI-powered natural language query.

Both the GP (fund manager) and LPs (investors) interact with this layer - each scoped to what they are permitted to see. Reporting and AI query are equal outputs of the same underlying data model.

The three apps

Submit - used by portfolio companies to connect their accounting system or upload financial documents. Claude extracts and standardises figures into Clavio's data model each quarter.

Partner/GP - used by the fund manager. Full view of all portfolio companies' standardised data, trend analysis, anomaly detection, and an AI query assistant ("Ask Clavio") for natural-language questions across any portfolio company's financials.

Investor/LP - used by the fund's investors. Fund-level reporting (IRR, MOIC, TVPI, DPI, capital account) and an AI query assistant scoped to LP-appropriate data - same intelligence layer as the GP, narrower permissions.

Your scope

The application front-end, database schema, authentication system, AI integration, file storage, and deployment pipeline will already be built and tested before you are engaged. Your scope is focused on four areas:

Your responsibility Security audit

Review the Row-Level Security (RLS) policies, authentication configuration, and secrets setup. Confirm they meet the standard expected for handling regulated financial data. Provide written sign-off before any real client data enters the system.

Your responsibility Account and secrets ownership

Take formal ownership of all production accounts (Supabase, Vercel, Anthropic, GitHub). Establish key rotation schedules for API credentials and document an incident-response runbook. Secrets must never appear in code or git history.

Your responsibility QuickBooks integration - Intuit registration

The OAuth 2.0 connection flow, token storage, periodic sync, and QuickBooks-to-Clavio field mapping will be built. However, the Intuit developer application must be registered and owned by a named individual under the company account on the Intuit Developer Portal (developer.intuit.com). You will also need to manage the Intuit app review process required before accessing real company data in production.

Your responsibility GDPR and compliance plumbing

The technical infrastructure supports GDPR compliance (data deletion, audit logs, regional storage). Your scope covers: data retention policies, data subject access request (DSAR) tooling, and DPA counter-signatures with sub-processors (Supabase, Anthropic, Vercel). Legal review of these documents is required before real client data enters the system.

5
Security and data privacy

The following concerns are addressed at the infrastructure level. Your audit should verify these are correctly implemented.

Concern Status Detail
AI vendor data training ✓ Handled Anthropic commercial API terms explicitly exclude data from model training. A DPA and Zero Data Retention (ZDR) agreement will be in place before any real financial data enters the system.
Data residency ✓ Configured Supabase production project is on eu-west-2 (London). All financial data stays within the UK/EU.
Cross-fund data isolation ✓ By design Row-Level Security (RLS) in Supabase enforces data isolation at the database layer. An LP or portfolio company query physically returns no rows from another fund's tables, regardless of application-layer behaviour. Your audit should include automated tests verifying this.
Document storage ✓ By design Files are stored in Supabase Storage (encrypted at rest, AWS S3-backed) with the same RLS policies as the database. Presigned URLs expire after a short window - no permanent public links.
Secrets management ✓ Vercel All secrets (Anthropic API key, Supabase service role key, etc.) are held in Vercel's encrypted environment variable store. They do not appear in code or git history.
